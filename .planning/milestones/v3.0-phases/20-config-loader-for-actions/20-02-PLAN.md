  <behavior>
    - derive_from_actions(actions: list[ActionEntry]) returns a dict/dataclass with:
      - gesture_modes: dict[str, str] mapping gesture_name -> "tap"/"hold_key" (from trigger state: static->tap, holding->hold_key, moving->tap, sequence->tap)
      - gesture_cooldowns: dict[str, float] for actions with cooldown overrides
      - activation_gate_bypass: list[str] of gesture names with bypass_gate=True
      - action_maps built via build_action_maps-style logic: per-hand Action dicts keyed by gesture_name
      - For hand:"both", action appears in both right and left maps
      - For hand:"left", action appears only in left map
      - For hand:"right", action appears only in right map
    - Test: 3 actions (static tap, holding hold, one with cooldown) -> correct gesture_modes, cooldowns
    - Test: action with bypass_gate -> appears in bypass list
    - Test: hand scoping produces correct per-hand maps
  </behavior>
  <action>
    Create derive_from_actions() function in config.py. Takes list[ActionEntry] (from plan 01), returns a DerivedConfig dataclass (or dict) with gesture_modes, gesture_cooldowns, activation_gate_bypass, right_actions (dict[str, Action]), left_actions (dict[str, Action]).

    For each ActionEntry:
    - Infer fire_mode from trigger state: TriggerState.STATIC -> FireMode.TAP, TriggerState.HOLDING -> FireMode.HOLD_KEY, TriggerState.MOVING -> FireMode.TAP. For SequenceTrigger, always FireMode.TAP.
    - Call parse_key_string(entry.key) to get modifiers + key.
    - Build Action(key_string=entry.key, fire_mode=..., gesture_name=entry.name, modifiers=..., key=...).
    - Place into right/left/both maps based on entry.hand.
    - Collect cooldown overrides into gesture_cooldowns dict.
    - Collect bypass_gate gesture names into activation_gate_bypass list.

    NOTE: gesture_swipe_mappings and compound action maps are NOT derived here -- moving triggers will use a new signal path in Phase 22, not the old compound fire. For now, moving-trigger actions go into the per-hand maps keyed by a composite key or a separate moving_actions map. Use the approach: separate maps for static/holding actions vs moving actions vs sequence actions, so ActionResolver (Phase 22) can look them up by signal type.

    Create a DerivedConfig frozen dataclass:
    ```python
    @dataclass(frozen=True)
    class DerivedConfig:
        gesture_modes: dict[str, str]
        gesture_cooldowns: dict[str, float]
        activation_gate_bypass: list[str]
        right_actions: dict[str, Action]
        left_actions: dict[str, Action]
    ```
  </action>
