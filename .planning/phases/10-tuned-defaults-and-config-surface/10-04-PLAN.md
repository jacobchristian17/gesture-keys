Debug session root cause (from .planning/debug/swipe-preempts-static.md):
THREE interacting problems make the 10-03 fix ineffective:
1. TIMING GAP: is_activating is reactive -- only True after smoother (2 frames) + debouncer transition. Swipe arms and fires in 3-4 frames on approach motion BEFORE any gesture is formed.
2. THRESHOLD SENSITIVITY: User config has min_velocity=0.15, min_displacement=0.03 (~2.5x more sensitive than defaults). Approach motion easily exceeds these.
3. PIPELINE RESET ON ARM: smoother.reset() and debouncer.reset() called when swiping transitions True, destroying in-progress static classification.

CRITICAL CONSTRAINT: Both __main__.py and tray.py have duplicated detection loops that MUST be modified identically.

DESIGN DECISION -- is_activating / hand-presence interaction:
The is_activating gate in both detection loops currently feeds `update(None)` to swipe_detector when the hand is physically present but a static gesture is being confirmed. The new _hand_present tracking inside SwipeDetector must NOT interpret these None calls as "hand left" -- otherwise when is_activating clears, the next non-None frame would trigger a spurious settling period.

SOLUTION: Add a `suppressed: bool = False` parameter to `SwipeDetector.update()`. When is_activating is True, callers pass `suppressed=True` with the real landmarks (or None if truly absent) instead of faking None. Inside update(), when suppressed=True, the method skips threshold/arming logic but does NOT clear _hand_present. This cleanly separates "hand physically absent" from "hand present but swipe suppressed by debouncer priority."
